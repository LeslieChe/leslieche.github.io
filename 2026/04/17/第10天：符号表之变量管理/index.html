从 0 型文法到 3 型文法

复习一下前文讲的形式化文法。

一个形式化文法通常由一个四元组定义：

N：非终结符集合。表示语法结构的中间变量，用尖括号或大写字母表示。

T：终结符集合：语言的基本符号，无法再分解，比如前面词法分析得到的各种 token。注意：非终结符和终结符没有交集。

P：产生式集合。描述替换规则，定义了如何从非终结符出发，逐步推导出合法的终结符号串。形式：α -> β（读作：α 能推导出 β）。α是一个包含至少一个非终结符的符号串。β是由终结符和非终结符组成的任意符号串（包括空串 ε）。

S：开始符号。指定一个非终结符号为开始符号，是整个推导过程的起点。

形式化文法是一个大家族，此家族有 4 个成员：0 型文法（无限制文法）、1 型文法（上下文有关文法）、2 型文法（上下文无关文法，CFG）和 3 型文法（正则文法）。

从 0 到 3，文法描述能力依次减弱。

0 型文法，产生式形如α -> β，其中α和β是任意符号串，但α不能为空串。任何计算机能计算的问题，它都能描述。但也因为过于强大，难以用于实践分析。

1 型文法，产生式形如αAγ -> αβγ。规则 A -> β只能在左侧上下文是 α、右侧上下文是 γ的情况下应用。描述能力弱于 0 型，但强于 2 型。可以描述像 {aⁿbⁿcⁿ | n ≥ 1}（相同数量的 a、b、c）这类上下文无关文法无法描述的语言。

2 型文法，产生式形如A -> β。规则的左侧必须只有一个非终结符，并且可以在任何上下文中被替换为β。它是实践中最有用、最广泛的一类文法，能高效地描述编程语言中大量的嵌套结构。

3 型文法，产生式形如A -> a或 A -> aB（右线性）。规则右侧最多只能有一个非终结符，并且必须出现在最左或最右。描述能力最弱。它描述的语言可以被有限状态自动机识别，常用于描述词法模式（如标识符、数字、关键字）。

前面我们为 csub 构造文法的时候，准确讲是构造 2 型文法(上下文无关文法） 。因为我们定义的产生式，左部仅有一个非终结符，并且可以在任何上下文中被替换为右部。

因为上下文无关，所以在语法分析“a = b;”的时候，无法回答变量 a 或者 b 是不是已经定义或声明，也无法回答 foo(a, b, c);的实参列表和函数定义时指定的形参列表是否一致。这些问题都牵扯到了上下文，是语法分析阶段做不了的。所以，必须做语义分析。

有的读者迫不及待，说我们赶快做语义分析吧。

符号表

在语义分析阶段，针对代码

a = b;

首先要查符号表，看看 a 和 b 这两个变量是否定义或声明，如果没有，那就报错。如果有定义或声明，还要看看类型是否兼容，假如 b 是指针， a 是 int，这样赋值肯定是不合法的。所以，当变量定义或者声明的时候，要把变量的名字、类型等信息加入符号表，这样才有据可查。

同理，遇到代码

foo(1,2);

也要查符号表，看看 foo 这个函数定义或者声明了没有，如果没有就报错。如果有定义或声明，还要看这个函数有几个参数，是不是 2 个，是不是都是整形。所以，遇到函数定义或者声明，要把函数的名字、返回类型、参数个数和类型等信息加入符号表。

再比如，遇到代码：

再回到函数 arr_init_list

第 551 和 566 行，调用 array_semantic_check，这个函数的作用是检查数组元素的类型是否和初始化数据的类型相匹配。这属于语义分析。

第 552 行的参数 INIT_BY_STRING 表示用字符串初始化字符数组。例如char my_str[] = "hello";

第 566 行的第二个参数 INIT_BY_LIST 表示用{…} 初始化数组。

第 553 和 567 行，如果语义检查通过，res 就是 true，这时候把 arr 添加到变量表，否则调用 var_destroy，释放内存。

由此可见，语义分析和符号表管理不是孤立的，而是交织在一起的。例如在把变量添加到符号表之前，需要检查语义是否正确，若正确才能把变量加入符号表。
