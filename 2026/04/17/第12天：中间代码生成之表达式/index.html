第12天：中间代码生成之表达式

第12天：中间代码生成之表达式

终于来到了中间代码生成环节。前面我们说过，符号表管理、语义分析和中间代码生成是纠缠在一起的三兄弟，所以本文还会看到他们三个难解难分的戏码。

有人问，为什么要生成中间代码呢？直接生成某个平台的机器码不好吗？我理解你这种“一步到位”的急切盼望，但是，我们不能一步到位。

现代编译器通常采用“前后端分离”的设计架构，这极大地提升了其灵活性和可移植性。简单来说，前端负责“理解”源代码，将其转化为一种中间表示；后端则负责“翻译”，将这种中间表示优化并转换为目标机器代码。

如果你追求一步到位，那就损失了灵活性和可移植性。比如现在我们直接生成 x86 平台的机器码，可是过几天想换成 ARM 平台，怎么办？那就要大修大改，甚至推倒重来。所以我们先生成中间代码，再把中间代码转换成目标机器代码。如果想换个硬件平台，那就再为这个硬件平台开发一个后端，或者使用现成的后端。

举个例子，如果要支持 M 种语言运行在 N 种机器上，需要编写 M * N 个完整的编译器；而如果利用中间代码，只需要 M 个前端 + N 个后端，大大降低了工作量和难度。

中间代码是一种通俗的说法，在编译原理中称为中间表示（Intermediate Representation，简称 IR）。

中间表示的形式有很多，其中两种最重要的中间表示形式是：

  • 树形结构，包括语法分析树和抽象语法树
  • 线性表示形式，特别是“三地址代码”

语法分析树和抽象语法树的区别在于前者严格遵循文法规则，完整记录语法推导过程的所有细节（包括终结符和非终结符）；后者仅保留对编译有实际意义的语法结构（如表达式、语句、声明等）。

三地址代码（Three-Address Code, TAC）

csub 使用的中间表示是三地址代码。

三地址代码（或者叫三地址码）得名于其指令格式——每条指令最多包含三个地址（即两个操作数和一个结果）。它的核心特点是结构简单、接近机器指令，是编译器优化和代码生成的关键桥梁。

三地址码有如下特征：

  1. 以指令为单位
  2. 每条指令只有有限数量的运算分量 (通常≤3)
  3. 支持基本数据类型及其运算
  4. 支持复合数据类型 (如数组、结构体) 及其操作
  5. 支持函数

运算分量指：

  1. 变量：源程序中变量
  2. 常量：源程序中出现或生成的常量
  3. 编译器生成的临时变量 (可与源程序变量等同视之)

三地址码的指令格式统一为：

operand1和 operand2 代表操作数，op 代表运算符，result 代表计算结果。

三地址码是抽象的概念，具体到代码实现该怎么做呢？这就要了解“四元式”了。

一个四元式有 4 个字段，分别是 op、arg1、 arg2、result；这四个字段分别和三地址码的 op、operand1、operand2、result 对应。

字段 op 包含一个运算符的内部编码（例如枚举类型）。

op 字段存放“+”，arg1 中存放 y，arg2 中存放 z，result 中存放 x

需要注意，对于像 x = y 这样的赋值语句， op 是 “=”，而对大部分其他双目运算符而言，赋值运算符是隐含表示的。

__copy_var_outline

为什么会有这个奇怪的函数，名字前面还有2个下划线

前面加 __ 是个人习惯，表明这是一个内部使用的函数。

此函数的作用是创建一个变量 v 的“模子”，准确讲，调用此函数后，得到的临时变量 var 仅仅拥有了 v 的类型；当 v 是基本类型时，var 的类型和 v 一样；如果 v 是数组或指针，则创建一个同类型的指针。

为什么当 v 是数组时要创建指针呢？

这是因为在 C 标准中，除去以下两种情况，数组的名字会自动转换为指向首个元素的指针（注意，不是左值）

第 2 行，如果 sizeof 的操作数是数组名字，那么它就代表这个数组。

第 3 行，如果数组名字作为 & 运算符的操作数，那么它也代表数组。

除去这两种，其他情况数组名字都会自动转换为指向首个元素的指针。

第 4 行是错误的写法，确切说，数组转换为指针类型的值，而不是转换为一个指针类型的变量。

ABI

ABI（Application Binary Interface，应用程序二进制接口）是程序编译成二进制机器码后，在运行时需要遵守的一套底层规则和约定。它定义了编译后的二进制程序如何与操作系统、库函数以及其他二进制程序模块进行交互。

一份完整的 ABI 覆盖很多内容，包括但不限于调用约定、可执行文件格式、符号修饰、基本类型大小与对齐、寄存器用途分类、栈帧布局等等。

为什么需要 ABI 呢？理由有很多，对于我们学习编译原理来说，有一条最重要的理由：让不同编译器生成的 .o 文件能链接到一起（或者说编译后的二进制代码可以互相交互）。

常见 ABI 标准

ABI 标准/家族 主要应用平台/架构 核心特点与说明
System V ABI 类 Unix 系统（Linux, macOS, BSD）的 x86-64 和 i386 架构 类 Unix 世界的基石，详细规定函数参数传递（如 x86-64 下用 RDI, RSI, RDX, RCX, R8, R9）、栈帧布局、数据对齐等
Microsoft ABI Windows 平台 Windows 生态的王者，64位时代统一为 Microsoft x64 Calling Convention（参数优先用 RCX, RDX, R8, R9 传递），与 System V ABI 不兼容
ARM ABI (AAPCS) ARM 架构（如移动设备、嵌入式系统） 包括 AAPCS（ARM 架构过程调用标准），规定参数通过寄存器（如 r0-r3）传递、栈使用规则等。Android 系统就基于此 。常见变体有 armeabi-v7a（32位）和 arm64-v8a（64位）

我们这节课只需要了解调用约定。

1972 年前后，Dennis Ritchie 在 PDP-11 上写 C 编译器时，机器寄存器很少（8 个通用寄存器），参数传递是用最朴素的办法：

  • 参数从右向左压，这样第一个参数在栈顶，printf 这种可变长参数函数就能顺着栈顶依次解析。
  • 调用者清理栈，因为调用者最清楚到底传了几个字节（尤其是变参函数）。
  • 函数返回时不用管栈，代码量最小。

这套做法在当时“足够好”，Unix 内核、 utilities 全部用它，源代码、目标文件、汇编接口自然形成事实标准。
后来人们才给这种“原始”做法起了个名字——cdecl，并把它固定成一种约定，以便不同编译器生成的目标文件能互相链接。

链接器层面的需求——“得给这种惯例起个名字”
70 年代末，Unix 开始移植到 VAX、68000，商用编译器（Lattice、Microsoft、Borland）也冒了出来。
不同编译器如果压栈顺序、清栈责任不一致，目标文件就无法静态链接。
于是各种“调用约定”被命名、标签化：

其中 cdecl 被 Microsoft、Borland、GCC 一致保留，唯一目的是“继续把老 C 代码链进来”。

  • 所有 Unix-like 系统的 ABI 默认就是 cdecl（System V AMD64 把它细化成“rdi, rsi, rdx, rcx, r8, r9 寄存器 + 栈”的扩展版，但仍兼容可变参部分）。
  • 任何用到 printf/scanf/open/... 的代码，只要参数列表可变，就必须走“调用者清栈”这一支，否则栈高度算不准。
  • 因此 GCC/Clang/MSVC 在 x86 32-bit 模式下都继续提供 __attribute__((cdecl)) / __cdecl，保证旧目标文件还能链接。

在 32位 x86 架构下，GCC 编译的 C 语言代码默认遵循 cdecl调用规则（C Declaration），比如 GCC 的 string.h标准库函数（如 strcpy, memcpy, strlen等）默认遵循 cdecl调用规则。

二是类似于 int a[3] = {1, 2, 3};

这种情况执行第 1038 行的 __gen_arr_init_by_list

__gen_arr_init_by_string 和 __gen_arr_init_by_list 就不具体讲了，留给读者分析。
