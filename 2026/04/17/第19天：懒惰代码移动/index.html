有读者说，如果把对 a + b 的计算前提到 B1 中，就像图 3 这样，是不是也可以呢？

​ ↑图3

当然可以。不过 LCM 的目标是让表达式的计算时刻尽量靠后（所以叫做“懒惰”）。为什么呢？

我们看 B1-B3-B4 这条路径。

图 3 在 B1 就计算了 a + b 的结果是 t， 变量 t 在使用前会一直留在寄存器中，如果计算 t 太早但是使用 t 太晚(可以想象 B3 中干了很多事情)，那么这个寄存器就要一直保留，不能用于其他计算，这是资源的浪费，毕竟处理器中的寄存器十分有限。

如果是 LCM　算法，如图 2，选择在 B３到 B4 的边上插入 t = a + b;

这样做的目的是尽量缩短从 t 被定值时刻到它被使用时刻的时间间隔，相当于减少寄存器的占用时间。

所以，把 a + b 的计算放到 B1中，会增加 t 对寄存器的使用时间。

懒惰代码移动遵循如下原则：仅在必要时移动代码，且尽量将计算向下移动，靠近使用点，而不是盲目上提到最早的可能位置。

模仿上面的做法，在基本块 B2 末尾增加一条指令可以吗？

不可以！因为 B2 也是 B4 的前驱，如果在基本块 B2 末尾增加一条指令，相当于在 B2 到 B4 的边上也增加了这条指令。

在基本块 B3 开头增加一条指令可以吗？

不可以！同理，会影响 B1 到 B3，相当于在 B1 到 B3 的边上也增加了这条指令。

在编译原理中，图 3 的红色边叫做关键边（Critical Edge）

关键边是指从一个具有多个后继的基本块（例如 B2）到另一个具有多个前驱的基本块（例如 B3）的边。

如何解决呢？

我们可以在 B2 到 B3 的边上插入一个空的基本块（红色虚线），也叫做合成块（synthetics）。把要增加的指令加入到这个基本块中。如此，就不会影响到其他路径。

龙书上的 LCM，为了简化算法，规定只在块的入口放置指令。

考虑到这个限制，我们再思考一遍刚才假设：根据优化结果，我们需要在 B2 到 B3 的边上增加一条指令。

如果是图 2，在 B3 入口插入这个指令，没有问题。

如果是图1，就不能简单地在 B3 入口插入这个指令了，而是需要在 B2 到 B3 的边上增加一个合成块。

所以可以得出结论：根据优化结果，假设需要在某条边上增加一条指令，并且规定只在块的入口放置指令。为了正确性，我们要关注具有多个（两个或以上）前驱的基本块，在它的每个前驱指向它的边上插入合成块。

有两个原因会导致这种情况：一是此基本块里的指令和要分析的表达式无关，为了突出重点，就不写了。

二是此基本块里面确实没有指令。有人说怎么可能，没有指令是怎么构成基本块的？我们的代码不是做了优化吗？活跃变量分析的时候，就把一些指令标记为死指令了。考虑极端情况，如果一个基本块里面的指令都被标记为死指令，就相当于这个块里面没有指令。

另外，合成块里面也没有指令，根据优化的结果，可能给里面添加指令。

第一关：被预期执行的表达式

先了解一下什么是“被预期执行”。

龙书 P412，如果从程序 P 点出发的所有路径最终都会计算表达式 b + c 的值，并且 b 和 c 在那时的值就是它们在点 p 上的值，那么我们说一个表达式 b + c 在程序点 p 上被预期执行。

请问上图中表达式 b + c 在 p 点是否预期执行？

答案是否。因为定义说“所有”路径，左边的路径确实预期执行 b + c ，但是右边的路径不会遇到 b + c。

把流图改一下，请问在 p 点表达式 b + c 是否预期执行？

答案是否。虽然所有路径都能遇到 b + c，但是请注意，右边先遇到 b = 3，这导致在表达式 d = b + c 中计算出的 “b + c”的值，并不一定是在 p 点计算出的“b + c”的值。我们说 b = 3 这条语句杀死了表达式 b + c;

换句话说，如果在 p 点 b + c 预期执行，那么你在 p 点计算 b + c 不是白忙活，这个计算结果在未来一定能用上，可以用于 p 点之后每条路径上首先遇到的那个 b + c；

请看上图，请问在 p 点表达式 b + c 是否预期执行？

答案是肯定的。

明白了上述原理，再看数据流分析的公式就好理解了。因为是预期执行，看的是未来，所以是逆向；因为是每条路径，所以交汇运算是交集。

传递函数： f(x) = e_use(B) | (x - e_kill(B))

e_use(B) 的意思是指令中使用的表达式，例如 b + c;

e_kill(B) 的意思是指令中杀死的表达式，比如 b = 3 杀死了表达式 b + c;

需要注意的是，对于表达式 j = j + 1;

e_use(B) = j + 1；

e_kill(B) = j + 1；

因为定义说 “b 和 c 在那时的值就是它们在点 p 上的值”，所以一旦遇到某个指令杀死了表达式，就需要从集合中排除掉这个表达式。这就是公式末尾有个减去 e_kill(B) 的原因。

对于每个指令，在初始化的时候要设置指令的 e_use(B) 和 e_kill(B) 集合；对于合成块或者空的基本块，e_use(B) 和 e_kill(B) 集合都是空集。

因为是逆向，所以从出口（B11）的 in 开始分析。我们关注表格中第二列的结果。

注意到 B8 的 in 是 0，说明表达式 “b + c” 在 B8 的入口没有被预期执行。理由很简单，B8 - B811- B11 这条路径上没有遇到 “b + c”；

再向上面看，经过了 B7 后， in 变成 1 了。

然后往上分成 2 条路径，左边在 B3 的入口还是 1，但是经过指令 80， 在 B2 的入口，值变成 0；

右边在 B5 的入口还是 1，在 B1 的出口变成 0（因为交汇运算是求交集）

请思考，“b + c”最早被预期执行的地方是哪里？

说白了就是计算“b + c”不白计算，而且越早计算越好。

蓝色的格子，都是“b + c”最早被预期执行的地方。

规律就是观察指令或基本块的 in 值，从 B1（入口）块开始，沿着执行路径观察，in 的值是在哪里从 0 变成 1 的，在 in 刚变成 1 的地方，就是我们要找的地方。

需要注意的是，“最早被预期执行”的“最早”，有点误导人，让我们误以为从入口到出口的某条路径上最多有一个最早被预期执行的地方。其实不然，请看下图：

指令29，指令 30 和 B13 的入口，都是“b + c”最早被预期执行的地方。

完善一下刚才的结论：从入口块开始，沿着执行路径观察，in 的值是在哪里从 0 变成 1 的，此指令或者基本块的入口就是我们要找的地方；如果后面又遇到了从 0 到 1 的跳变，那么这个地方也是最早被预期执行的地方。

为什么我们关注最早预期执行的地方？

请看下图。

如果在 p3 点计算 t = b + c, 那么 d2 和 d3 就不用再计算 b + c 了(替换为 d2 = t; d3 = t;)相当于减少了一次计算。

如果在 p2 点计算 b + c, 那么 d1、d2 和 d3 就不用再计算 b + c 了，相当于减少了两次计算。

但是不能在 p1 点计算,在 p1 点， b + c 没有被预期执行。

所以，为了消除更多的冗余，我们应该找到最早预期执行的地方。

第二关：可用表达式

什么是可用表达式？

龙书 P391 给的定义是：如果从流图入口点到达程序点 p 的每条路径都对表达式 x + y 求值，且从最后一个这样的求值之后到 p 点的路径上没有再次对 x 或 y 赋值，那么 x + y 在点 p 上可用（available）。

请问上图中 b + c 在 p 点可用吗？

答案是不可用，因为定义说“每条路径”，左边求值了，但是右边没有求值。

请问上图中 b + c 是否在 p 点可用？

答案是不可用，因为右边先有 b + c，然后遇到了 b = 3;杀死了 b + c；

请问 b + c 是否在 p 点可用？

可用。

第一，从流图入口（B0）到达程序点 p 的每条路径都对表达式 b + c 求值;

第二，右边的路径，虽然出现了 b = 3，但是从第二次求值(e = b + c)到 p 点的路径上没有杀死 b + c;

总结一下，“可用”相当于“已计算且未失效”。

不过，LCM 的可用表达式和上面的 available 还是有区别的，LCM 的是 “will be available”。

什么意思呢？想象在所有 anticipated.in = 1 的基本块入口都插入表达式 t = b + c;然后做一个可用表达式数据流分析。

传递函数是 f(x) = (anticipated(B).in | x) - e_kill(B)

龙书（P415）对这个公式的解释是：

一个表达式在一个基本块的出口处可用的条件有 2 个：

1、下列条件之一成立

（1）在入口处可用

（2）在基本块的入口处所预期执行的表达式集合中（如果选择在这里计算它，则它就成为可用的了）

2、没有被这个基本块杀死

我认为如果（2）成立，那么第 2 条是必然成立的，因为如果表达式被杀死了，在入口处就不可能预期执行。如果读者不理解为什么要分析可用表达式，我们举个例子来说明。

第三关：可后延表达式

前文说过，为了减少寄存器的占用时间，应该把表达式的计算尽量推后。

先看左边，第四列，B3 中绿色的格子，因为 B3 是 earliest 且它没有使用 b + c，所以 postponable.out 是 1

如果一个块的 postponable[B].in = 1，我们就说可以推迟到这个块或者说这个块在可后延的集合中。

有读者会疑惑为什么左边不能推迟到 B7，这是因为如果这样做，当代码执行右边分支的时候，B7 会引入冗余计算，因为 B5 已经计算过了。

左右分支分别代表了两种情况：

1、右边的 B5，表达式被使用了，所以失去了后延的机会

2、左边 B407 的 postponable.out = 1，但是 B607 的 postponable.out = 0，因为交汇运算是求交集，所以 B7 postponable.in 是 0，导致不能后延到 B7；

上面就是最后位置（latest）的计算公式。

首先，block 要在 earliest 或可后延集合中（公式的第一行，我把这成为“基本条件”），例如上图中的 B3,B4,B407,B5

其次，还要满足下面 2 个条件之一。

(1) block 使用了表达式（公式第二行的 EUse_b）

(2) b 存在一个后继，不在 earliest 和可后延集合中(公式中对 b 的后继求交集再取反)

如果它的所有后继都满足条件，求交集得 1，再取反得 0，那它就不满足条件。如果它的后继至少有一个不满足条件，求交集是 0，再取反得 1，那它就满足条件。

总结

1、插入合成块，这一步很重要，因为有时候需要在边上增加指令；

2、被预期执行的表达式分析，防止引入不必要的计算；

3、将要可用的表达式分析，为了结合 2 找到 earliest

4、可后延表达式分析，为了找到 latest

5、可用表达式分析，清理无用代码

6、插入和替换
