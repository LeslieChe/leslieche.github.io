第17天：复写传播

复写传播（Copy Propagation）是编译器优化中的一种重要技术，它通过消除不必要的复写指令（如 x = y）来提升程序性能。

第 5~7 行，都是复写指令。

基本原理

复写传播的数据流分析框架如下：

1、数据流方向：前向，我们需要沿着代码执行的方向分析每一条指令。

2、边界值和初始值：边界集合是空集；初始值集合是所有复写指令的全集。

3、交汇运算：当基本块有多个前驱的时候，基本块的 in 值是多个前驱 out 值的交集。

4、传递函数：s.out = (s.in - s.kill) ∪ s.gen（这里的减号表示差集)

对于 1 和 3，前面的课程已经讲过；

对于 2 和 4，估计读者会困惑；没有关系，马上就讲。

递归出口就是遍历了 in 集合中的所有表达式，发现没有任何一个可以替换，这时候返回 var 本身，就是第 436 行。

第 430 行需要解释一下，保不齐代码中有 a = a； 这样的复写表达式，查找 a 的源头，是 a, 再查找 a 的源头，是 a, 哎呀，死循环了！所以要特殊处理，一旦发现，就 break;

有读者会问，会不会构成环，这样也会无限查找下去。

例如， c = b 的 in 集合中有下面 3 个复写表达式：

b = x; x = y; y = b;

（1）b = x; 说明 b 的源头是 x

（2）x = y; 说明 x 的源头是 y

（3）y = b; 说明 y 的源头是 b

继续查找 b 的源头，又回到（1），死循环了。

实际上不可能，为什么呢？

假设集合里面已经有了 y = b 和 x = y， 此时要加入 b = x； 这会杀死 y = b ；所以这 3 个复写表达式不能共存。

之所以没有 IR_OP_LEA 是因为这是取地址运算，不能替换。

例如 a = b; p = &a;

不能因为 a 的源头是 b, 就把 p = &a; 替换成 p = &b;
