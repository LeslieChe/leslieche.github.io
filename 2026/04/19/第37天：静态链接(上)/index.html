第37天：静态链接(上)

第37天：静态链接(上)

终于到了最后一关——链接器。过了这关，我们的编译系统就大功告成了。
为什么要链接？
在编写大型程序时，我们不可能把所有代码写在一个文件里，一般都是按照功能划分模块，每个模块对应一个或多个 .c；
编译器的工作是把每个 .c 编译成 .o；
链接器的工作是把这些 .o 按某些规则拼接在一起，形成一个可以 run 的可执行文件。

链接可分为静态链接和动态链接。
这两种方式有何不同呢？
简单来说，它们的区别在于“链接动作发生的时间”和“库代码被复制的方式”。
1、静态链接
在编译阶段，链接器将程序调用的库函数代码，直接完整地复制到最终的可执行文件中。
就像你在写论文时，把参考书上的原文直接抄写到了你的论文里。这导致的结果是你的论文非常厚，但好处是读者不需要再去翻参考书了。
2、动态链接
在编译阶段，链接器只在可执行文件中记录所需的库文件名称和符号信息，而不复制库代码。真正的链接动作推迟到程序加载时或运行时进行，由操作系统完成。
就像你在写论文时，只是在文中标注了“详见《xxx》第XX页”，并没有把原文抄下来。这样你的论文很薄。但是，读者在阅读你的论文时，需要随时查阅你引用的文献，否则就不知道你说的是什么。

我们要写的链接器比较简单，只考虑静态链接。
整个链接过程可以分成下面的步骤:

  1. 收集每个 .o 的信息
    可以理解为上节课的逆过程，上节课我们是把 .o 的各部分（用各种结构体、链表描述）打包到 elf_32 格式中，这节课是把 elf_32 格式的内容还原出来（装进各种结构体和链表）。

  2. 符号解析
    这是一个“连连看”的过程。例如 a.o 里面定义了 foo 函数，b.o 里面调用了 foo 函数，那么链接器就认为“对上了”。
    如果同一个全局符号在多个 .o 中都有定义，链接器就抛出符号重复定义的错误（为了简化问题，这里忽略强弱符号的裁决），链接失败；如果某个全局符号被引用了，但是在所有的 .o 中都找不到它定义，则抛出未定义符号错误，链接失败。

  3. 查找入口点
    查找入口点（Finding the Entry Point） ，是指链接器在生成可执行文件时，确定程序第一个被执行的指令在哪里的过程，第一条指令所在的地址就是程序的 “入口点”。操作系统加载程序后，会直接跳转到该地址开始执行。
    注意，程序的入口点 ≠ main函数。
    main 函数是我们编写的代码的入口，但在 main 执行前，需要先执行初始化代码（如初始化全局变量、解析命令行参数、设置栈帧等）；入口点是初始化代码的起始地址，初始化完成后，才会调用 main 函数；
    简单说：入口点是操作系统与程序的 “对接点”，main 是用户代码的 “起点”。

  4. 地址分配
    简单说，链接器的地址分配（Address Assignment/Layout） 是指链接器将多个目标文件（.o）的 section（如.text、.data 等）合并，并为每个段分配最终的虚拟地址（Virtual Address） 和文件偏移（File Offset）

  5. 符号地址解析
    当地址分配完成后，每个段的虚拟地址就确定了。段的虚拟地址确定了，段内定义的符号的地址就确定了：用段基地址加上符号的偏移地址，得出符号的最终地址。
    另外，也是很重要的一点，就是看看这个符号是否被其他文件引用，如果是，则要把此符号的最终地址“传递”给那个文件的符号表。

  6. 符号重定位
    重定位开始前，必须完成两件事：
    1） 符号解析完成：所有 “引用符号” 已绑定到对应的 “定义符号”
    2） 地址分配完成：每个 “定义符号” 已获得唯一的绝对虚拟地址
    上面两个步骤，我们已经在第 2、4 步骤完成了。
    接下来就是读取每个 .o 的重定位表，根据公式计算修正后的地址，并把修正后的地址写到合适的位置。具体细节我会在讲解代码的时候说清楚。

  7. 组装可执行文件并输出
    这是最后一步，和上节课输出目标文件相比，可执行文件多了一个程序头表，少了重定位表。

有的读者问，为什么要填充到 0x1000、0x2000 呢？这不是浪费硬盘吗？
这是一个好问题。

我们先看看这个 a.out 加载到内存中是什么样子。请看右边的图。
根据程序头表，我们发现链接器给这 3 个段分配的虚拟地址分别是 0x08048000，0x08049000，0x0804a000
程序头表的最后一列，Align 是 0x1000，什么意思呢？
段加载到虚拟内存时，其起始虚拟地址必须是 align 值的整数倍；是的，这既是硬件的要求，也是操作系统的强制约束。
所以，地址都是 0x1000 的倍数；但是为什么从 0x08048000 开始，而不是从其他数字开始（例如 0x07650000）？
对于 Linux 系统 32 位可执行文件，默认的基地址是 0x08048000；

从 CPU 的角度讲，在内存层面要遵循权限隔离原则
页是权限的最小单位：CPU 的 MMU（内存管理单元）只能按页（通常是 4KB）来设置 Read/Write/Execute 属性。
不能混住：如果一个 Segment（如只读的代码段）和另一个 Segment（如可读写的数据段）挤在同一个物理页里，系统就无法给这个页设置单一的权限。
所以，每个 Segment 的起始虚拟地址必须是页大小的整数倍（即 0x1000 对齐）。
以上只解释了内存中的段要按页对齐，并没有解释为什么磁盘上的段也要对齐。

Linux 加载程序时，并不是用 read() 把文件读进内存，而是用 mmap() 把磁盘文件直接映射到虚拟内存。
为了让这种映射效率最高，硬件要求文件内的偏移量（File Offset）和内存中的虚拟地址（Virtual Address）必须对齐。
公式：Virtual_Address % Page_Size == File_Offset % Page_Size
为了满足这个公式，链接器在布局各个段时，如果段的大小没凑够 4KB，就会在磁盘文件里填充 0。虽然这浪费了一点磁盘空间，但换取了极快的启动速度（内核只需要修改页表，不需要搬运数据）。

为了缩短查找时间，我们定义一个哈希表。
sym_defs_tab 里面仅加入定义的符号

此函数用来判断是否有符号重复定义，参数应该传入 sym_defs （导出符号）链表的头节点；
第 383 行，遍历 sym_defs 链表，第 385，计算符号的 key，key 值决定映射到哈希表的哪一个桶；
第 386 行，遍历哈希表中对应的桶，如果第 388 行为真，说明此符号已经在哈希表中了，第 390 行报错，重复定义；
例如 a.o 定义了变量 hello，b.o 也定义了变量 hello，这肯定不行；
如果遍历完桶上的所有节点，都没有遇到同名的符号，说明没有重复定义，第 397 行，把它加入哈希表；
以上函数执行后，哈希表 sym_defs_tab 中就是所有 .o 中定义的符号。

是否未定义

除了判断符号是否重复，还要判断符号是否未定义

此函数的参数应该传入 sym_refs 链表的头节点。

第 410 行，遍历这个链表，拿出每一个被引用的符号。计算 Key 值，在哈希表中查找，如果找到，填写它的 prov 字段，也就是定义此符号的文件。

如果在哈希表中查不到，第 422 行报错，符号未定义。
