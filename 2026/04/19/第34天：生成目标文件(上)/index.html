既然要生成目标文件，那就要明确我们生成的目标文件长什么样子。
上节课的目标文件结构示意图已经给出了方向。不过有些细节，还需要确认一下。
命令是：
nasm -f elf32 test.asm -o test_nasm.o
readelf -a test_nasm.o
test_nasm.o 的总大小是 42864 字节,十六进制是 0xA770;
readelf 命令，正如它的名字一样，是展示 elf 文件信息的;
-a 就是 —all，表示打印所有的信息;

首先，会输出文件头

然后，输出段表

这里有 8 个段表项。

再然后，输出了重定位表

最后是符号表

我们先看段表，也叫节头表。

⬆段表举例图

之前讲过 elf 文件的格式，这里不再赘述。我们要讨论的是 .o 都由哪些部分组成，以及排列顺序是什么？
虽然 elf 格式并没有规定节头表中每个表项的 off 字段要依次递增，但是 nasm 很讲究，观察 off 这一列，发现是递增的。

还有一个问题，索引 [1] 是代码段(.text)，从偏移 0x180 开始，那么 0x180 之前是什么？
根据文件头，Start of section headers: 64 (bytes into file)
可以知道偏移 64 字节开始就是段表了。

再根据 Size of section headers: 40 (bytes)
Number of section headers: 8
可以知道段表占了 320 字节。
64 + 320 = 384 = 0x180；
所以，我们确定 0x180 之前是段表。
根据段表最后一个条目，也就是 .rel.data 段，偏移是 0xa760, 大小是 8，所以结尾就是 0xa768，再填充 8 个字节的零，刚好是 0xa780；
我是怎么知道填充了 8 个字节的零？
hexdump -C -s 0xa768 test_nasm.o
-C 意思是 “规范的十六进制 + ASCII 字符同步显示”;
-s 0xa768 表示跳过前面 0xa768 个字节;

你瞧，确实是 8 字节的 0;

需要说明的是：段表不一定要紧跟着文件头，也可以放在 .o 的最后，如果你愿意，甚至可以把段表塞在某两个段之间。
为了方便验证（拿我们生成的 .o 和 nasm 生成的 .o 比较），我们模仿 nasm 的布局。
有读者说，那我们是不是先输出一个文件头，然后需要什么内容就用 fwrite 函数往文件里写。
道理没有错，不过调用 fwrite 是最后的步骤。我们可以先构造出图中每个模块，例如有几个段表项，每个段表项的字段填什么值，代码段的内容是什么，待所有字段和内容敲定后，再往文件里面写。

我们的节头表，就按照这个顺序从 0 开始排列表项。

有几点要说明：

  1. shdr_order[] 表示节头表里面表项的顺序，不是段的排列顺序；段的物理排列顺序不一定和节头表里面表项的顺序一致。只是 nasm 比较讲究，让段表中每个表项的 off 字段依次递增，这导致的结果就是段的物理分布和节头表表项的顺序一致。
  2. 第一个必须是空表项，空表项的 size 是 0，所以在文件中不占用物理空间。也就是说目录上有，但是没有对应的内容。
  3. 也不是每个文件都拥有这些段。例如有的文件没有代码，只有全局变量，那就没有 .text；还有的文件没有 .rel.data；
  4. 如果某个段不存在怎么处理？我们的处理方法是段表中依然有对应的表项，但是 size 字段是 0；

下面要解决的问题是如何获得代码段和数据段的大小。
在汇编器做语法分析的时候，我们设置了一个段计数器，每当遇到指令或者数据定义，这个计数器就会增加。

第 913 和第 917 行，text_sec_size_g、data_sec_size_g 都是全局变量，初始值为 0；
当遇到 section .xxx 的时候，就会调用第 919 行的函数，第 923~926，根据段的名字，把计数器的值赋给 text_sec_size_g 或者 data_sec_size_g；
语法分析结束后，代码段和数据段的大小就拿到了；
其他文件通过调用 get_text_sec_size() 和 get_data_sec_size() 获得这 2 个值。

st_size 填 0 就行，前文说过了。
第 455 行，遍历哈希表，添加局部符号;
第 460 行，遍历哈希表，添加全局符号;
可能读者会好奇，为什么要遍历两次，分开添加局部和全局符号？
因为在 ELF 规范中，符号表（Symbol Table）的排列顺序是有规定的，
根据 System V ABI 标准，符号表项必须遵循以下顺序：
所有具有 STB_LOCAL（局部）绑定属性的符号都必须排在所有具有 STB_GLOBAL（全局）或 STB_WEAK 属性的符号之前。
我们不考虑 STB_WEAK 属性的符号，所以只有局部和全局两种符号。

段表项里面有一个字段叫 sh_info，对于符号表，记录符号表内最后一个局部符号的索引+1；对于重定位表，指向被修补的节的索引；
链接器（Linker）在处理重定位或符号解析时，通常只关心全局符号，所以只需要查看符号表的段表项，取出 sh_info，从它开始就是全局符号的索引；
如果排列顺序是随意的，那么链接器必须遍历整个符号表并检查每一个符号的 st_info 属性，效率会大大降低。

第 3 个参应该传入 shdr_order，此数组记录段表项的顺序；
第 4 个参数是 shdr_order 数组的元素个数；
第 139 行，求 sh_size， 等于符号的个数 * 每个符号条目的大小。
第 140 行，获得 .strtab 段的索引，赋值给 sh_link，这是因为对于符号表，sh_link 指向与之关联的字符串表 (.strtab) 的索引。因为符号表中每个符号的名字，要在字符串表中查询偏移。
